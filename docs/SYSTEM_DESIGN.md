## 3. Hard System Design Problems & Solutions

### 3.1 ACID Transactions for Multi-Step Operations

**Problem:** Resolving an incident involves updating 4 tables. If any step fails, the system must be in a consistent state.

**Solution: Prisma `$transaction`**

```javascript
// src/services/IncidentService.js
async resolveIncident(incidentId, staffId, note, photo) {

  // 1. Load and hydrate domain object
  const incidentRow = await prisma.incident.findUnique({
    where: { id: incidentId },
    include: { assignedTo: true }
  })
  const incident = this.incidentRepo.toEntity(incidentRow)

  // 2. Run domain logic (State pattern — no DB involved)
  incident.resolve(note, photo)   // throws if invalid (photo missing, wrong state etc.)
  const staffId = incident.assignedToId

  // 3. Persist all changes atomically
  await prisma.$transaction(async (tx) => {

    // Update incident status + resolution fields
    await tx.incident.update({
      where: { id: incidentId },
      data: {
        status: 'RESOLVED',
        resolutionNote: note,
        resolutionPhoto: photo,
        resolvedAt: new Date()
      }
    })

    // Append to audit trail (never update existing rows)
    await tx.incidentStatusLog.create({
      data: { incidentId, status: 'RESOLVED', changedById: staffId, note }
    })

    // Decrement staff active task count atomically
    await tx.user.update({
      where: { id: staffId },
      data: { activeTaskCount: { decrement: 1 } }
    })

    // Create feedback notification
    await tx.notification.create({
      data: {
        recipientId: incident.creatorId,
        type: 'FEEDBACK_REQUEST',
        title: 'Rate the resolution',
        body: `How well was ${incident.incidentNumber} resolved?`,
        incidentId
      }
    })
  })
  // If ANY of the 4 above fails → PostgreSQL rolls ALL back
  // Incident stays IN_PROGRESS, staff count unchanged, no phantom notification

  // 4. Fire domain events AFTER successful persist
  await this.publisher.publish('INCIDENT_RESOLVED', { incident })
}
```

**Interview answer:** "I wrap the resolution flow in a Prisma transaction. PostgreSQL guarantees atomicity — if the notification insert fails after the status update, the entire transaction rolls back. The incident stays IN_PROGRESS with no partial state. This is ACID compliance — impossible to guarantee this cleanly with MongoDB without multi-document transactions."

### 3.2 Concurrent Assignment — Row-Level Locking

**Problem:** Two admins simultaneously try to assign the same OPEN incident. Without locking, both succeed — the incident has two "first assignments."

**Solution: `SELECT FOR UPDATE` inside Prisma transaction**

```javascript
async assignIncident(incidentId, staffId, adminId) {
  return await prisma.$transaction(async (tx) => {

    // Lock the row — second concurrent transaction waits here
    const rows = await tx.$queryRaw`
      SELECT id, status, "assignedToId"
      FROM "Incident"
      WHERE id = ${incidentId}
        AND status = 'OPEN'
      FOR UPDATE
    `

    if (rows.length === 0) {
      throw new IncidentNotAssignableError(
        'Incident is not open or does not exist'
      )
    }

    // Safe to proceed — we hold the lock
    await tx.incident.update({
      where: { id: incidentId },
      data: { assignedToId: staffId, status: 'IN_PROGRESS' }
    })

    await tx.user.update({
      where: { id: staffId },
      data: { activeTaskCount: { increment: 1 } }
    })

    await tx.incidentAssignment.create({
      data: { incidentId, assignedToId: staffId, assignedById: adminId, strategy: 'manual' }
    })

    await tx.incidentStatusLog.create({
      data: { incidentId, status: 'IN_PROGRESS', changedById: adminId, note: 'Manual assignment' }
    })
  })
  // Lock released when transaction commits
}
```

**Interview answer:** "`SELECT FOR UPDATE` acquires a row-level exclusive lock. If Admin A and Admin B both try to assign the same incident, one gets the lock and the other waits. When Admin A's transaction commits, Admin B's `SELECT FOR UPDATE` returns zero rows because status is now IN_PROGRESS — not OPEN. Admin B gets IncidentNotAssignableError. Database prevents the race condition — no application-level mutex needed."

### 3.3 SLA Escalation — Crash-Safe with BullMQ

**Problem:** Guarantee escalation fires at the SLA deadline even if the server crashes mid-processing.

**Solution: BullMQ delayed jobs + idempotency check**

```javascript
// Schedule on incident creation:
const job = await slaQueue.add(
  'escalate-incident',
  { incidentId: incident.id, escalationLevel: 1 },
  {
    delay: incident.sla.deadlineAt - Date.now(),
    jobId: `sla:${incident.id}:1`,   // deterministic — enables deduplication
    attempts: 3,
    backoff: { type: 'exponential', delay: 1000 }
  }
)

// Store jobId on incident for cancellation
await prisma.incident.update({
  where: { id: incident.id },
  data: { slaJobId: job.id }
})

// Worker — idempotent processing:
async function processEscalation(job) {
  const { incidentId } = job.data

  // Use transaction to prevent race between worker retries
  await prisma.$transaction(async (tx) => {
    const incident = await tx.$queryRaw`
      SELECT id, status, "slaIsEscalated"
      FROM "Incident"
      WHERE id = ${incidentId}
      FOR UPDATE
    `

    if (!incident[0]) return   // incident deleted
    if (incident[0].slaIsEscalated) return   // already escalated (retry scenario)
    if (incident[0].status === 'RESOLVED') return  // resolved before deadline

    await tx.incident.update({
      where: { id: incidentId },
      data: {
        slaIsEscalated: true,
        slaEscalatedAt: new Date(),
        status: 'ESCALATED'
      }
    })

    await tx.incidentStatusLog.create({
      data: { incidentId, status: 'ESCALATED', note: 'SLA breach — auto escalated' }
    })
  })

  await notificationService.notify(escalationTarget, 'ESCALATION', incident)
}

// Cancel on resolution:
if (incident.slaJobId) {
  const job = await slaQueue.getJob(incident.slaJobId)
  if (job) await job.remove()
}
```

### 3.4 Panic Button — Speed Over Consistency

**Problem:** Sub-200ms broadcast to all online security officers. Normal incident creation (validation chain + DB writes + assignment) takes ~300ms.

**Solution: Emit first, persist asynchronously**

```javascript
// src/realtime/handlers/panic.handler.js
socket.on('panic_trigger', async ({ lat, lng }) => {
  const reporter = await getUserFromSocket(socket)

  // Step 1: Emit IMMEDIATELY — no DB, no async (< 5ms)
  io.to('role:SECURITY').emit('panic_alert', {
    reporterId: reporter.id,
    reporterName: reporter.name,
    lat, lng,
    timestamp: new Date().toISOString()
  })

  // Step 2: Confirm to reporter immediately
  socket.emit('panic_confirmed', {
    message: 'Security officers have been notified.'
  })

  // Step 3: Persist asynchronously — non-blocking
  setImmediate(async () => {
    try {
      await incidentService.createEmergencyIncident({
        creatorId: reporter.id,
        panicLat: lat,
        panicLng: lng,
        broadcastedAt: new Date()
      })
    } catch (err) {
      logger.error('Emergency incident creation failed after panic broadcast', err)
      // Broadcast already happened — acceptable failure
      // Can be reconciled manually by admin
    }
  })
})
```

**Interview answer:** "The panic broadcast happens before the incident is persisted. I intentionally chose speed over consistency here. Security officers are notified in under 200ms — before any database write. The incident record is created asynchronously. If creation fails, the broadcast still happened. For a panic button, getting help to the student is the only priority."

### 3.5 Dashboard Caching with Targeted Invalidation

**Problem:** Admin dashboard aggregation queries are expensive. The dashboard is viewed far more often than data changes.

**Solution: Redis cache-aside via Proxy pattern**

```javascript
// src/infrastructure/repositories/CachingIncidentProxy.js
class CachingIncidentProxy {
  constructor(realRepo, redis) {
    this.real = realRepo
    this.redis = redis
  }

  async getDashboardStats(deptId) {
    const key = `cache:dashboard:dept:${deptId}:stats`
    const cached = await this.redis.get(key)
    if (cached) return JSON.parse(cached)

    const stats = await this.real.getDashboardStats(deptId)
    await this.redis.setex(key, 300, JSON.stringify(stats))
    return stats
  }

  // Called by CacheInvalidator observer on incident status change
  async invalidate(deptId) {
    await this.redis.del(`cache:dashboard:dept:${deptId}:stats`)
    await this.redis.del('cache:dashboard:global:stats')
  }
}
```

### 3.6 Rate Limiting — Redis Sliding Window

```javascript
// Lua script — atomic, no race condition
const SLIDING_WINDOW_SCRIPT = `
local key = KEYS[1]
local now = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local limit = tonumber(ARGV[3])

redis.call('ZREMRANGEBYSCORE', key, 0, now - window)
local count = redis.call('ZCARD', key)
if count >= limit then return 0 end
redis.call('ZADD', key, now, now .. math.random())
redis.call('EXPIRE', key, math.ceil(window / 1000))
return 1
`

// Role-based limits:
const LIMITS = {
  STUDENT:        { 'POST /incidents': { requests: 5,    windowMs: 3600000 } },
  FACULTY:        { 'POST /incidents': { requests: 20,   windowMs: 3600000 } },
  ADMIN:          { 'GET /analytics':  { requests: 1000, windowMs: 3600000 } },
  unauthenticated:{ 'POST /auth/login':{ requests: 5,    windowMs: 900000  } }
}
```

## 4. Authentication Flow

```
POST /auth/login { email, password }
          │
          ▼
Find User by email (Prisma)
          │
          ├── lockedUntil > now? → 423 ACCOUNT_LOCKED
          │
          ▼
bcrypt.compare(password, passwordHash)
          │
          ├── FAIL →
          │     increment failedLoginCount
          │     if >= 5: set lockedUntil = now + 15min
          │     return 401 INVALID_CREDENTIALS
          │
          └── SUCCESS →
                reset failedLoginCount = 0
                generate accessToken (JWT, 15min, ACCESS_SECRET)
                generate refreshToken (UUID v4)
                hash refreshToken → bcrypt → store in User.refreshTokenHash
                return { accessToken, refreshToken, user }

POST /auth/refresh { refreshToken }
          │
          ▼
Find User (must have refreshTokenHash set)
          │
          ▼
bcrypt.compare(refreshToken, refreshTokenHash)
          │
          ├── MISMATCH → TOKEN REUSE DETECTED
          │     Clear User.refreshTokenHash (invalidate all sessions)
          │     return 401 TOKEN_REUSE_DETECTED
          │     (both attacker and real user must re-login)
          │
          └── MATCH →
                generate new accessToken
                generate new refreshToken
                hash new refreshToken → update User.refreshTokenHash
                old token is now invalid (hash replaced)
                return { accessToken, refreshToken }
```

## 5. Domain Object Hydration from PostgreSQL

This is the bridge between Prisma rows and domain classes:

```javascript
// src/infrastructure/repositories/IncidentRepository.js

const STATE_MAP = {
  OPEN:        () => new OpenState(),
  IN_PROGRESS: () => new InProgressState(),
  RESOLVED:    () => new ResolvedState(),
  ESCALATED:   () => new EscalatedState(),
  REOPENED:    () => new ReopenedState()
}

const INCIDENT_MAP = {
  MAINTENANCE:     (data) => new MaintenanceIncident(data),
  SECURITY:        (data) => new SecurityIncident(data),
  INFRASTRUCTURE:  (data) => new InfrastructureIncident(data),
  EMERGENCY:       (data) => new EmergencyIncident(data),
  CLEANLINESS:     (data) => new MaintenanceIncident(data),
  OTHER:           (data) => new Incident(data)
}

toEntity(row) {
  const incident = INCIDENT_MAP[row.category](row)
  incident.setState(STATE_MAP[row.status]())
  incident.sla = SLAFactory.fromRow(row)
  return incident
}

toDocument(incident) {
  return {
    status: incident.state.getName(),   // 'IN_PROGRESS' etc.
    slaWindowHours: incident.sla.windowHours,
    slaDeadlineAt: incident.sla.deadlineAt,
    slaIsEscalated: incident.sla.isEscalated,
    // ... rest of fields
  }
}
```

PostgreSQL stores strings. Domain reconstructs objects. Zero conflict between LLD and SQL.
